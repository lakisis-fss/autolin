# 리니지 클래식 메모리 직접 리딩 및 이미지 인식 이원화 가이드

본 문서는 리니지 클래식 프로젝트에서 메모리 직접 리딩 방식과 이미지 인식 방식의 독립성을 보장하고 두 기술의 근본적인 원리와 오독을 방지하는 정밀 필터링 구조에 대해 상세히 정리한 학습 문서입니다.

---

## 1. 두 가지 데이터 획득 방식의 이해

컴퓨터가 실행 중인 게임에서 실시간으로 캐릭터의 상태 정보를 얻는 방법에는 크게 두 가지가 있습니다. 하나는 사람처럼 눈으로 화면을 보고 판단하는 이미지 인식 방식이고, 다른 하나는 뇌 속의 신경 신호를 직접 가로채는 것과 같은 메모리 직접 리딩 방식입니다.

### 1.1 이미지 인식 방식
*   **원리**: 컴퓨터 비전 기술을 사용하여 모니터 화면에 출력된 게임 그래픽 중 체력바나 숫자 텍스트 영역을 캡처한 뒤, 이미지 분석 인공지능 모듈을 거쳐 글자를 해독합니다.
*   **비유**: 학생이 시험 문제를 풀기 위해 칠판에 적힌 문제를 눈으로 직접 바라보고 자신의 공책에 옮겨 적는 과정과 같습니다. 칠판 글씨가 너무 작거나 빛이 반사되면 잘못 받아적는 오독이 발생할 수 있습니다.
*   **장점**: 게임 클라이언트에 아무런 흔적을 남기지 않고 순수하게 화면만 캡처하여 동작하므로 가장 안전하고 투명합니다.
*   **단점**: 게임 화면에서 다른 창이 캐릭터 스탯 창을 가리거나 글꼴 해상도가 깨지는 등 외부 환경의 영향을 크게 받습니다.

### 1.2 메모리 직접 리딩 방식
*   **원리**: 게임이 실행되면서 컴퓨터의 임시 저장 장치에 올려놓은 캐릭터의 체력, 레벨, 경험치, 무게 데이터가 기록된 물리 메모리 주소를 포인터 체인을 통해 직접 리딩합니다.
*   **비유**: 칠판을 보는 대신, 선생님의 교탁 위에 놓인 공식 성적표 장부를 직접 열어서 캐릭터의 점수를 그대로 복사해 오는 과정과 같습니다.
*   **장점**: 화면이 가려지거나 흔들려도 전혀 영향을 받지 않으며, 소수점 아래 여섯 자리까지의 극도로 정밀한 경험치처럼 화면에 다 표현되지 않는 참값까지 100% 오차 없이 정확하게 읽어올 수 있습니다.
*   **단점**: 게임 클라이언트의 가상 메모리 공간에 직접 연결하므로 오프셋 구조가 변경되거나 가방의 힙 구조체가 재할당되는 셔플링이 발생할 때 메모리 주소를 다시 찾아내야 하는 까다로운 추적 기술이 필요합니다.

## 2. 하이브리드 연동의 위험성과 이원화 설계의 필요성

기존 시스템은 이미지 인식으로 알아낸 값들을 메모리 스캐너의 힌트로 사용하거나, 메모리에서 알아낸 초정밀 수치로 이미지 인식 결과 그리드를 덮어씌워 보정하는 이른바 하이브리드 연동 방식을 채택하고 있었습니다.

하지만 이러한 상호 간섭 설계는 연쇄적 오류의 덫에 빠지기 쉽습니다. 예를 들어, 포만감 100%라는 글자를 이미지 판독기가 10%로 잘못 해석하여 메모리 스캐너에게 힌트로 제공하면, 메모리 스캐너는 멀쩡히 작동하던 정확한 캐시를 버리고 10%에 해당하는 메모리 주소를 찾기 위해 엉뚱한 힙 공간을 헤집어 놓게 됩니다. 결국 한 곳의 사소한 이미지 판독 오류가 전체 모니터링 시스템을 마비시키는 도미노 현상이 일어나게 됩니다.

따라서 이번 설계 개선에서는 두 개의 데이터 파이프라인을 완전히 독립적으로 작동하도록 분리하였습니다. 이미지 판독기는 화면에 보이는 대로만 솔직하게 기록하고, 메모리 스캐너는 스스로의 물리적 특징과 규칙을 이용해서 참값을 찾도록 구조적 이원화를 보장했습니다.

## 3. 핵심 기술 구현 및 최적화 기법

### 3.1 숫자 전용 정밀 필터링 기법
이미지 판독기가 판독해야 하는 대부분의 영역이 영문 알파벳 노이즈에 쉽게 오염된다는 한계를 극복하기 위해, 판독 대상이 오직 숫자로만 구성된다는 긍정적 전제를 기반으로 정밀 필터링 알고리즘을 장착했습니다.

```python
# 영문 폰트 오독 방지용 강제 숫자 매핑 테이블의 예시
replace_map = {
    "S": "7", "s": "7",
    "A": "0", "O": "0", "o": "0", "D": "0",
    "l": "1", "I": "1", "i": "1", "t": "1",
    "B": "8", "b": "8",
    "G": "9", "g": "9",
    "Z": "2", "z": "2"
}
```

이 필터는 저해상도 게임 폰트 특성상 1과 I를 헷갈려하거나 0과 O를 구별하지 못해 발생하는 오류를 완벽하게 통제합니다. 변환된 문자열에서 숫자와 소수점, %, - 기호를 제외한 모든 무의미한 문자를 깨끗이 날려버림으로써 완벽한 수치 데이터를 보장하게 됩니다.

### 3.2 배타적 다중 프로필 검출 알고리즘
서로 다른 게임 실행 환경이나 레벨 변화에 따라 캐릭터의 내부 메모리 구조체 오프셋이 바뀔 때, 주소 오독을 원천 방지하기 위해 엄밀한 다중 프로필 우선순위 전환 방식을 적용했습니다.

*   **동작 원리**: 특정 레벨(예: 11레벨)의 진짜 메모리 주소 영역을 최우선적으로 엄밀하게 선별 대조합니다. 이 조건이 완벽히 만족될 때 즉시 프로필을 고정 확정하며, 찌꺼기 주소나 우연히 일치하는 쓰레기 변수로 인해 다른 프로필(예: 14레벨 프로필)로 오인되는 오독 확률을 100% 배제합니다.
*   **세션 리셋**: 프로필의 물리적 전환이 감지되는 실시간 임계 시점에 기존 가방 캐시 주소를 완전히 파괴하고 초기화함으로써, 이전 캐릭터 세션의 정보가 엉켜 오염을 퍼뜨리는 현상을 원천 방어합니다.

### 3.3 물리적 상대 거리를 이용한 고밀도 동적 스캔 (16바이트 거리 제한 룰)
외부의 이미지 힌트 없이도 메모리 스캐너가 실시간으로 재할당되는 가방 구조체를 탐색하기 위해, 데이터의 타당성 범위와 메모리상에서의 상대적 거리를 분석하는 2차원 기하학적 스캔 기법을 설계했습니다.

*   **무게의 물리 범위**: 0 이상 100 이하의 정수값
*   **포만감의 물리 범위**: 80 이상 1200 이하의 정수값 (100% = 1000)
*   **상대적 거리의 극단적 축소 제한 (16바이트)**: 리니지 클래식 내부 데이터 구조 상, 무게(Weight) 변수와 포만감(Food) 변수는 캐릭터 정보 구조체의 바로 옆자리(연속된 주소 공간)에 나란히 배치되어 있어 두 오프셋의 거리가 최대 16바이트를 넘지 않습니다.

이 오프셋 간격 제한을 스캔 조건으로 장착함으로써, 가방 구조체 내부가 아닌 힙 메모리 전체에 흩어져 있는 무작위 인덱스 수량이나 쿨타임, 고유 아이템 ID와 같은 쓰레기 변수들이 무게 1% 혹은 포만도 45% 등으로 오결정되어 오프셋 캐시를 마비시키는 버그를 수학적으로 완벽하게 통제하였습니다.
