# 리니지 클래식 포만도 메모리 추적 성공 기록

## 1. 개요
리니지 클래식 클라이언트의 메모리에서 '포만도'와 '무게' 값을 찾아내어 실시간 대시보드에 동기화하는 작업을 진행했습니다. 그 과정에서 값의 자료형(Data Type)에 대한 아주 중요한 비밀을 풀었고, 완벽하게 동기화에 성공했습니다.

## 2. 발생했던 문제점 (유령 변수의 함정)
처음에는 포만도가 40%일 때, 메모리 어딘가에 숫자 `40` 또는 `400` 같은 정수형(Integer)으로 저장되어 있을 것이라고 가정했습니다.
하지만 아무리 정수로 검색을 해도 값이 제대로 떨어지지 않거나, 스캔에 실패하여 수치가 0%로 뻗어버리는 문제가 발생했습니다.

## 3. 소름 돋는 발견 (Float의 비밀)
범위를 좁혀가며 메모리를 뒤진 결과, 게임 클라이언트가 포만도를 정수가 아닌 **소수점(Float, IEEE-754 32bit)** 형태로 저장하고 있다는 사실을 발견했습니다!

예를 들어, 화면에 `40%`라고 표시될 때, 메모리 내부의 실제 숫자는 `1109634581` 이라는 알 수 없는 정수 형태로 보였습니다. 하지만 이 숫자를 32비트 소수점(Float)으로 변환하면 정확히 **`40.920005...`** 가 됩니다.
즉, 캐릭터가 사탕을 먹거나 시간이 지날 때마다 포만도는 1% 단위가 아니라 0.001% 단위의 소수점까지 아주 정밀하게 변동하고 있었으며, 화면 UI에는 소수점을 자른 정수만 보여주고 있었던 것입니다.

## 4. 해결 방법 및 패치 내역
1. **소수점 전용 스캐너 제작**: `find_food_float.py`를 제작하여 0.0 ~ 100.0 사이의 소수점 값만 추적하도록 만들었습니다.
2. **AOB 스캐닝 방식 변경**: `mem_state_reader.py` 엔진 내부에서 값을 판독하고 유효성을 검사할 때, 기존의 `read_int` 대신 `read_float`를 사용하도록 정밀하게 패치했습니다.
3. **황금 지문 추출**: 12개로 좁혀진 동일한 포만도 값들 중, 완벽히 오차가 없는 진짜 주소를 찾아내어 AOB(Array of Bytes) 16바이트 지문을 생성하고 `mem_state_reader.py`에 적용했습니다.

## 5. 내일 진행할 목표 (무게 메모리 추적)
포만도와 마찬가지로, **'무게(Weight)' 역시 소수점(Float)으로 저장되어 있을 확률이 99.9%** 입니다.
오늘 포만도 스캔에서 사용했던 "증가(1) / 유지(2) / 감소(3)" 필터링 기법과 `find_food_float.py`의 플롯(Float) 스캔 방식을 응용하여, 내일은 무게에 대한 완벽한 AOB 지문을 찾아내고 대시보드에 적용할 예정입니다.

## 6. 결론
결과적으로 포만도가 오차 없이 대시보드에 즉각 반영되는 것을 확인했습니다. 데이터가 화면에 보이는 그대로 저장되어 있지 않고 수학적으로 인코딩되어 있다는 해킹의 기본 원리를 완벽하게 돌파한 기념비적인 작업이었습니다.
