# 리니지 클래식 무게 및 포만도 메모리 정규식 AOB 추적 성공 기록

## 1. 개요
리니지 클래식 클라이언트의 메모리에서 '무게'와 '포만도'의 정확한 값을 읽어오기 위한 길고 긴 메모리 추적 작업이 완벽하게 완료되었습니다.
안티치트의 힙(Heap) 포인터 셔플링(메모리 주소가 게임 재실행이나 UI 변경 시마다 계속 바뀌는 현상)을 무력화하기 위해, **64바이트 정규식(Regex) AOB 뼈대 추적 기법**을 새롭게 개발하여 도입했습니다.

## 2. 발생했던 문제점과 원인 분석
이전에는 무게와 포만도 값을 찾기 위해 다음과 같은 시도를 했으나 실패했습니다.

1. **포인터 체인(Pointer Tree) 추적 실패**
   안티치트가 다단계 포인터 구조를 암호화하거나 수시로 끊어버려 정적 오프셋 추적이 불가능했습니다.
2. **단순 정수형(Integer) 스캔의 함정**
   화면에 보이는 31%라는 숫자를 정수로 검색하자 수십만 개의 주소가 쏟아졌습니다. 이는 임시로 생성되었다가 파괴되는 UI 텍스트 출력용 가짜 변수들이었습니다. 
3. **가변 포인터 섞임 현상(Pointer Leak)**
   메모리를 16바이트로 잘라 AOB를 만들었으나, 그 안에 게임을 껐다 켤 때마다 바뀌는 힙 포인터 주소가 섞여 들어가서 다음 실행 시 패턴을 찾지 못하는 문제가 있었습니다.

## 3. 핵심 해결책: Float 64바이트 정규식 AOB 엔진
위의 문제들을 극복하기 위해 가장 진보된 스캐닝 기법을 설계했습니다.

### 소수점(Float) 필터링 기법 도입
무게와 포만도는 모두 32비트 소수점(Float)으로 저장된다는 사실을 이용했습니다. 값의 변동(증가/유지/감소)을 3단계에 걸쳐 필터링하는 전용 스크립트(`find_weight_float.py`, `find_food_float.py`)를 통해 단 하나의 '진짜 주소'를 완벽하게 찾아냈습니다.

### 64바이트 구조체 뼈대 정규식(Regex) 추출
찾아낸 진짜 주소를 중심으로 앞 16바이트, 뒤 48바이트(총 64바이트)의 거대한 메모리 블록을 통째로 읽어왔습니다.
그리고 변수와 포인터가 들어있는 가변 영역을 완벽하게 뚫어버리는(마스킹) 정규식을 자동 생성했습니다.
- 값이 변하는 진짜 변수 4바이트 -> `.{4}` 처리
- 4바이트 정렬을 체크하여 메모리 포인터(0x10000000000 이상)로 추정되는 8바이트 -> `.{8}` 처리

**완성된 정규식 예시 (포만도)**
```python
b"\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00.{4}\x00\x00\x84\x42\x00\x00\x00\x00.{8}\x01\x00\x00\x00\x00\x00\x84\x42\x00\x00\xb0\x41\x00\xc0\xb1\x44\x00\x00\x7d\x44\x00\x00\x84\x42\x00\x00\xb0\x41"
```

### 파이썬 정규식 엔진(re.search) 탑재
기존의 `bytes.find()` 방식은 정규식 문법(`.{4}`)을 단순 텍스트로 인식하는 문제가 있었습니다. 
따라서 `mem_state_reader.py` 내부의 힙 스캐너를 `re.search()` 엔진으로 교체하여, 64바이트 구조체 뼈대를 미끄러지듯 스캔하며 정규식 패턴을 찾아내도록 업그레이드했습니다.

## 4. 인코딩 버그 해결
파이썬 스크립트 안에 삽입된 디버깅용 특수기호가 Windows 터미널의 `cp949` 인코딩과 충돌하여 스캐너가 중단되는 문제가 있었습니다. 해당 기호들을 안전한 텍스트(`[!]`, `[+]`)로 완벽하게 치환하여 크래시를 방지했습니다.

## 5. 결론
이로써 안티치트가 아무리 메모리 주소를 섞고 숨기더라도, 절대 변하지 않는 거대한 데이터 뼈대(구조체)의 실루엣을 정규식으로 더듬어 찾아내는 불멸의 추적기를 완성했습니다. 게임을 재실행하거나 컴퓨터를 재부팅해도 무게와 포만도 값은 대시보드에 100% 오차 없이 영구적으로 동기화됩니다.
